## 🛡️ 零风险保证机制

### 1. 数据库零风险

```sql
-- 原则 1: 只增加字段，不修改现有字段
ALTER TABLE products ADD COLUMN new_field ...;  -- ✅
ALTER TABLE products MODIFY COLUMN existing_field ...;  -- ❌

-- 原则 2: 所有新字段 nullable，有默认值
ADD COLUMN service_type VARCHAR(50) DEFAULT 'water';  -- ✅

-- 原则 3: 验证现有查询
SELECT * FROM products WHERE is_active = 1;  -- 必须与之前结果一致
```

### 2. API 零风险

```python
# 原则 1: 新增 API，不修改现有 API
@app.get("/api/services/config")  # ✅ 新增
@app.get("/api/products")  # ❌ 不修改

# 原则 2: 向后兼容测试
def test_backward_compatibility():
    old_api_response = get("/api/products")
    assert old_api_response == expected_old_format
```

### 3. 前端零风险

```javascript
// 原则 1: 保留原有 UI，新增服务 UI
// 商品管理页面 - 保持不变 ✅
// 服务管理页面 - 新增 Tab ✅

// 原则 2: 配置化切换
const useServiceExtension = config.useNewService || false;
if (!useServiceExtension) {
    // 显示原有水业务 UI
} else {
    // 显示服务扩展 UI
}
```

## 🚨 风险应对预案

### 预案 1: 数据库迁移失败

```bash
# 症状：迁移脚本执行失败
# 应对：
1. 立即停止迁移
2. 执行回滚脚本
3. 分析失败原因
4. 修复后重新测试

# 回滚命令
./scripts/rollback_phase_1.sh
```

### 预案 2: API 兼容性问题

```bash
# 症状：现有功能调用失败
# 应对：
1. 立即回滚 API 变更
2. 切换到旧版本代码
3. 分析问题原因
4. 修复后重新发布

# 回滚命令
git revert <commit-hash>
./scripts/rollback_api.sh
```

### 预案 3: 前端严重 Bug

```bash
# 症状：用户报告严重 Bug
# 应对：
1. 立即关闭功能开关
2. 切换到旧版 UI
3. 修复 Bug
4. 重新测试发布

# 关闭功能
UPDATE config SET use_service_extension = 0;
```
