## 一、需求背景

### 1.1 用户痛点

| 痛点 | 描述 | 影响 |
|------|------|------|
| 无法查看历史预约 | 用户看不到过去的预约记录 | 体验差，缺乏历史追踪 |
| 取消规则不明确 | 不知道什么时候可以取消 | 用户困惑，投诉率高 |
| 随意取消 | 会议临近时取消，造成资源浪费 | 会议室空置，影响他人使用 |
| 缺少取消审核 | 重要会议取消无审核流程 | 管理混乱 |

### 1.2 业务目标

1. **提升用户体验**: 清晰的预约历史记录，透明的取消规则
2. **降低空置率**: 合理的取消限制，减少资源浪费
3. **规范管理**: 分级取消审核机制

## 二、取消规则设计

### 2.1 时间窗口策略

```
时间轴: ────────────────────────────────────────▶
        │            │           │         │
      预约时      前24小时     前2小时    开始时间
        │            │           │         │
        ├────────────┼───────────┼─────────┤
        │            │           │         │
      随时可取消   免费取消   限时取消   不可取消
```

### 2.2 取消规则矩阵

| 预约状态 | 距开始时间 | 用户权限 | 操作 | 说明 |
|---------|-----------|---------|------|------|
| **待确认** (pending) | 任意 | ✅ 完全权限 | 直接取消 | 未确认的预约，随时可取消 |
| **已确认** (confirmed) | ≥ 24小时 | ✅ 完全权限 | 直接取消 | 充足提前量，免费取消 |
| **已确认** (confirmed) | 2-24小时 | ⚠️ 有限权限 | 申请取消 | 需填写原因，管理员审核 |
| **已确认** (confirmed) | < 2小时 | ❌ 无权限 | 不可取消 | 时间过近，不可取消 |
| **进行中** (active) | - | ❌ 无权限 | 不可取消 | 会议进行中，不可取消 |
| **已完成** (completed) | - | ❌ 无权限 | 已结束 | 历史记录，不可操作 |
| **已取消** (cancelled) | - | ❌ 无权限 | 已取消 | 已取消状态 |

### 2.3 状态判断逻辑

```javascript
function canUserCancel(booking) {
    // 1. 已取消或已完成，不可再次取消
    if (['cancelled', 'completed'].includes(booking.status)) {
        return { allowed: false, reason: '该预约已结束' };
    }
    
    // 2. 进行中，不可取消
    if (booking.status === 'active') {
        return { allowed: false, reason: '会议进行中，不可取消' };
    }
    
    // 3. 待确认状态，随时可取消
    if (booking.status === 'pending') {
        return { allowed: true, type: 'direct', reason: '预约待确认，可随时取消' };
    }
    
    // 4. 已确认状态，根据时间判断
    if (booking.status === 'confirmed') {
        const now = new Date();
        const bookingTime = new Date(`${booking.booking_date}T${booking.start_time}`);
        const hoursBefore = (bookingTime - now) / (1000 * 60 * 60);
        
        if (hoursBefore >= 24) {
            return { allowed: true, type: 'direct', reason: '提前24小时以上，可免费取消' };
        } else if (hoursBefore >= 2) {
            return { allowed: true, type: 'request', reason: '需申请取消，管理员将审核' };
        } else if (hoursBefore > 0) {
            return { allowed: false, reason: '距开始时间不足2小时，不可取消' };
        } else {
            return { allowed: false, reason: '会议已开始，不可取消' };
        }
    }
    
    return { allowed: false, reason: '未知状态' };
}
```

## 四、交互流程

### 4.1 直接取消流程（≥24小时）

```
用户点击"取消预约"
    ↓
弹出确认框
    ↓
显示取消规则说明
    ↓
选择取消原因
    ↓
确认取消
    ↓
调用取消API
    ↓
更新状态为"已取消"
    ↓
释放时间段
    ↓
发送取消通知
    ↓
显示成功提示
```

### 4.2 申请取消流程（2-24小时）

```
用户点击"申请取消"
    ↓
弹出申请框
    ↓
显示审核说明
    ↓
必须填写取消原因
    ↓
提交申请
    ↓
状态更新为"待审核"
    ↓
通知管理员
    ↓
管理员审核
    ├─ 通过 → 取消成功
    └─ 拒绝 → 恢复原状态
```

## 五、技术实现

### 5.1 前端实现要点

1. **时间计算**: 实时计算距开始时间的小时数
2. **按钮状态**: 根据规则动态显示/禁用按钮
3. **表单验证**: 取消原因必填验证
4. **用户体验**: 清晰的状态提示和引导

### 5.2 后端API设计

**取消预约接口**:
```
PUT /api/meeting/bookings/{booking_id}/cancel

Request Body:
{
    "cancel_reason": "临时有事无法参加",
    "cancel_detail": "客户临时取消会议"
}

Response:
{
    "success": true,
    "message": "预约已取消",
    "data": {
        "booking_id": 1,
        "status": "cancelled",
        "cancelled_at": "2026-04-03T10:30:00",
        "refund_amount": 800.00
    }
}
```
