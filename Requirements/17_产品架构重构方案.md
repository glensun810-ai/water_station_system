## 一、现状诊断与问题分析

### 1.1 当前系统全景图

```
Portal (入口页) — 静态卡片网格，仅做链接跳转，无业务功能
         │                    │                    │
         ▼                    ▼                    ▼
┌──────────────┐    ┌──────────────┐    ┌──────────────┐
│  水站管理     │    │  会议室预定   │    │  VIP 餐厅    │
│ Service_     │    │ Service_     │    │ Service_     │
│ WaterManage  │    │ MeetingRoom  │    │ Dining       │
│              │    │              │    │              │
│ + water-admin│    │ + meeting-   │    │ (早期阶段)    │
│   (独立管理)  │    │   admin      │    │              │
│              │    │   (独立管理)  │    │              │
└──────────────┘    └──────────────┘    └──────────────┘
```

### 1.2 核心问题清单

| # | 问题类别 | 具体表现 | 影响等级 |
|---|---------|---------|---------|
| 1 | **入口空心化** | Portal 仅为静态链接卡片，无业务数据、无个性化、无待办聚合 | 严重 |
| 2 | **管理端碎片化** | water-admin 和 meeting-admin 各自独立，管理员需切换不同系统 | 严重 |
| 3 | **用户端割裂** | 每个服务独立入口，用户无法统一查看"我的服务" | 严重 |
| 4 | **功能重复建设** | 登录、通知、用户管理等通用功能在各模块重复实现 | 中等 |
| 5 | **导航不一致** | 水站用侧边栏+底部Tab，会议室用顶部Tab+底部Tab，模式不统一 | 中等 |
| 6 | **跨服务能力缺失** | 无法跨服务查看账单、统计、通知 | 中等 |
| 7 | **服务扩展困难** | 新增服务（茶歇、保洁、接送）无统一接入框架 | 高 |
| 8 | **首页信息密度低** | 各模块首页仅展示本模块数据，缺乏全局视角 | 中等 |

## 二、目标产品架构设计

### 2.1 架构设计原则

1. **统一入口，分层服务**: Portal 升级为功能型首页，承载跨服务聚合能力
2. **模块自治，接口统一**: 各服务模块保持业务独立性，但遵循统一的接入规范
3. **用户视角优先**: 从"服务分类"转向"用户场景"组织功能
4. **管理端一体化**: 合并所有管理后台为统一 Admin 平台
5. **渐进式演进**: 保持现有功能可用，逐步迁移，零停机过渡

### 2.2 三层架构模型

```
L1: 平台层 (Portal)
    ├── 统一首页
    ├── 消息中心
    ├── 用户中心
    └── 服务中心
         │
         ▼
L2: 服务层 (各服务模块)
    ├── 饮用水服务 (完整业务)
    ├── 会议室服务 (完整业务)
    ├── VIP餐厅服务 (完整业务)
    ├── 茶歇服务 (规划中)
    ├── 保洁服务 (规划中)
    └── 接送服务 (规划中)
         │
         ▼
L3: 管理层 (Unified Admin)
    ├── 全局看板
    ├── 服务管理 (按服务分Tab)
    ├── 用户管理
    └── 系统设置
```

### 2.3 用户角色定义

| 角色 | 定义 | 核心场景 |
|------|------|---------|
| **普通员工** | 园区企业员工 | 领水、预约会议室、预定餐厅、查看个人记录 |
| **办公室管理员** | 办公室/部门负责人 | 管理本办公室领水、审批本部门预约 |
| **园区运营管理员** | 平台运营人员 | 管理所有服务资源、审批、财务结算、数据统计 |
| **超级管理员** | 系统管理员 | 用户管理、系统配置、服务接入管理 |

## 三、首页（Portal）功能设计

### 3.1 首页定位

首页不再是静态链接卡片，而是**个性化工作台**，核心定位：

> "让用户一眼看清今天该做什么、我的服务状态如何、快速进入常用服务"

### 3.2 首页功能架构

```
首页（Portal）
├── 顶部区域：个人工作台
│   ├── 用户问候 + 日期
│   ├── 个人数据概览（本月领水、本月预约、待结算金额）
│   └── 消息通知铃铛（未读数量角标）
│
├── 核心区域：待办与提醒
│   ├── 待办事项卡片
│   │   ├── 待结算通知（来自水站）
│   │   ├── 待审批预约（来自会议室，仅管理员）
│   │   ├── 即将到期的预约提醒
│   │   └── 库存预警（仅管理员可见）
│   └── 快捷操作入口
│       ├── 快速领水
│       ├── 预约会议室
│       ├── 预定餐厅
│       └── 查看我的记录
│
├── 服务入口区：服务矩阵
│   ├── 常用服务（用户自定义排序）
│   │   ├── 饮用水（显示库存状态）
│   │   ├── 会议室（显示今日可预约状态）
│   │   ├── VIP餐厅（显示今日包间状态）
│   │   ├── 茶歇服务
│   │   ├── 保洁服务
│   │   └── 接送服务
│   └── 全部服务（展开更多）
│
├── 数据看板区（仅管理员可见）
│   ├── 今日服务概览（各服务今日使用量）
│   ├── 本月收入趋势
│   └── 资源利用率排行
│
└── 底部区域
    ├── 我的服务记录（跨服务最近记录列表）
    └── 帮助中心 / 使用指南
```

### 3.3 首页功能归属决策矩阵

| 功能 | 归属 | 决策理由 |
|------|------|---------|
| 个人数据概览 | 首页 | 跨服务聚合，用户需要全局视角 |
| 待办事项聚合 | 首页 | 跨服务待办统一处理，减少遗漏 |
| 消息通知中心 | 首页 | 全局通知入口，各服务通知统一汇聚 |
| 服务快捷入口 | 首页 | 替代现有静态卡片，增加状态预览 |
| 我的服务记录（跨服务） | 首页 | 用户需要统一查看历史 |
| 快速领水 | 首页快捷入口 -> 跳转模块 | 首页提供快捷入口，实际操作在模块内 |
| 预约会议室 | 首页快捷入口 -> 跳转模块 | 首页提供快捷入口，实际操作在模块内 |
| 领水登记 | 水站模块 | 完整业务表单，保留在模块内 |
| 会议室预约表单 | 会议室模块 | 完整业务表单，保留在模块内 |
| 日历视图 | 会议室模块 | 专业视图，保留在模块内 |
| 结算管理 | 各服务模块 | 各服务结算逻辑不同，保留在模块内 |
| 资源管理（库存/会议室/包间） | 各服务模块 | 专业管理功能，保留在模块内 |
| 用户管理 | 统一用户中心 | 跨服务用户管理，建议独立模块 |
| 系统设置 | 管理后台 | 管理员功能，不面向普通用户 |

## 四、各服务模块功能规划

### 4.1 饮用水服务模块

**定位**: 办公室饮用水领用登记、库存管理、结算统计

**用户端功能**:
- 领水登记（选择办公室、产品、数量、领取人、时间）
- 我的记录（记录列表、筛选、批量结算、导出）
- 我的预付（余额概览、订单列表、充值入口）
- 我的优惠券（可用列表、使用记录）

**管理端功能**（合并入统一Admin）:
- 数据看板（水站维度）
- 领水记录管理
- 结算管理
- 产品与库存管理
- 预付账户管理
- 优惠券管理
- 办公室管理
- 回收站

**调整建议**:
- 保持现有功能完整
- services.html、bookings.html、packages.html 功能合并入主管理端
- 用户端保持独立 H5 页面（扫码场景）

### 4.2 会议室服务模块

**定位**: 会议室在线预约、时段管理、冲突检测、订单管理

**用户端功能**:
- 会议室列表/预约（用户类型选择、房间卡片、预约表单）
- 日历视图（月/周/日视图切换、房间筛选、拖拽预约）
- 我的预约（预约列表、支付提交、取消预约）

**管理端功能**（合并入统一Admin）:
- 数据看板（会议室维度）
- 会议室管理（CRUD、启用/禁用）
- 预约管理（筛选、审批、批量操作）
- 审批中心
- 财务结算
- 统计报表
- 系统设置

**调整建议**:
- 保持现有 3 页用户端结构（预约/日历/我的）
- 导航模式已较完善（顶部Tab + 移动端底部Tab）
- 管理端合并入统一 Admin

### 4.3 VIP 餐厅服务模块

**定位**: 商务宴请预定、包间管理、用餐服务

**用户端功能**:
- 在线预定（包间选择、套餐选择、日期时段、人数联系人）
- 我的预定（需新增：预定列表、预定详情、取消预定）

**管理端功能**（合并入统一Admin）:
- 包间管理
- 套餐管理（需完善增删改功能）
- 预约管理（确认/取消）

**调整建议**:
- 当前仅有单页预定表单，需补充"我的预定"功能
- 管理端功能尚不完整，需完善套餐管理
- 管理端合并入统一 Admin

### 4.4 未来服务模块接入规范

**统一接入要求**:
- 用户端标准页面结构：服务列表页、预约/登记表单、我的服务记录
- 管理端标准Tab：服务资源管理、预约/订单管理、数据统计
- 接入配置：服务类型定义、图标+主题色、单位列表、预约规则配置

## 五、统一管理后台设计

### 5.1 现状问题

当前存在两套独立管理后台：
- water-admin/admin.html — 水站管理（侧边栏 8 个 Tab）
- meeting-admin/admin.html — 会议室管理（侧边栏 7 个 Tab）

**问题**:
- 管理员需要在不同系统间切换
- 用户管理、通知等通用功能重复建设
- 无法跨服务查看全局数据
- 新增服务需要新建管理后台

### 5.2 统一 Admin 架构

```
统一管理后台
├── 全局看板（首页）

│   │   └── 库存/资源预警
│   ├── 快捷操作

├── 饮用水管理
│   ├── 领水记录
```

## 十、风险评估与应对

| 风险 | 影响 | 概率 | 应对措施 |
|-----|------|------|---------|
| 跨服务API聚合复杂 | 高 | 中 | 分步实现，先做核心服务聚合 |

## 十一、总结

### 核心改造点

1. **首页升级**: Portal 从静态卡片升级为个性化工作台
2. **管理端统一**: 合并 water-admin 和 meeting-admin 为统一 Admin
3. **服务模块保持自治**: 各服务模块用户端保持独立，管理端统一
4. **跨服务能力**: 首页聚合待办、通知、记录、数据看板

### 预期效果

- 用户：一站式查看所有服务状态，减少遗漏
- 管理员：统一入口管理所有服务，效率提升
- 系统：架构清晰，新服务接入标准化
- 运营：全局数据视角，决策更科学
