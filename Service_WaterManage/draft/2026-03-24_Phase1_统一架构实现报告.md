#### 1.2 表结构详情

**user_account - 用户账户表**
```sql
CREATE TABLE user_account (
    id INTEGER PRIMARY KEY,
    user_id INTEGER UNIQUE NOT NULL,
    balance_credit FLOAT DEFAULT 0.0,    -- 信用余额（先用后付）
    balance_prepaid FLOAT DEFAULT 0.0,   -- 预付余额（先付后用）
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
    updated_at DATETIME DEFAULT CURRENT_TIMESTAMP
)
```

**account_wallet - 账户钱包表**
```sql
CREATE TABLE account_wallet (
    id INTEGER PRIMARY KEY,
    user_id INTEGER NOT NULL,
    product_id INTEGER NOT NULL,
    wallet_type VARCHAR(20) NOT NULL,    -- 'credit' 或 'prepaid'
    available_qty INTEGER DEFAULT 0,     -- 可用数量
    locked_qty INTEGER DEFAULT 0,        -- 锁定数量
    total_consumed INTEGER DEFAULT 0,    -- 累计消费
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
    updated_at DATETIME DEFAULT CURRENT_TIMESTAMP,
    UNIQUE(user_id, product_id, wallet_type)
)
```

**settlement_batch - 结算批次表**
```sql
CREATE TABLE settlement_batch (
    id INTEGER PRIMARY KEY,
    batch_no VARCHAR(50) UNIQUE NOT NULL,
    user_id INTEGER NOT NULL,
    transaction_ids TEXT NOT NULL,       -- JSON 字符串
    total_amount FLOAT NOT NULL,
    status VARCHAR(20) DEFAULT 'pending',
    confirmed_at DATETIME,
    confirmed_by INTEGER,
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
    updated_at DATETIME DEFAULT CURRENT_TIMESTAMP
)
```

**transactions_v2 - 统一交易记录**
```sql
CREATE TABLE transactions_v2 (
    id INTEGER PRIMARY KEY,
    user_id INTEGER NOT NULL,
    product_id INTEGER NOT NULL,
    wallet_id INTEGER,
    quantity INTEGER DEFAULT 1,
    unit_price FLOAT NOT NULL,
    actual_price FLOAT NOT NULL,
    mode VARCHAR(20) DEFAULT 'credit',   -- 'credit' 或 'prepaid'
    wallet_type VARCHAR(20) DEFAULT 'credit',
    status VARCHAR(20) DEFAULT 'pending',
    settlement_batch_id INTEGER,
    discount_desc VARCHAR(200),
    free_qty INTEGER DEFAULT 0,
    note TEXT,
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
    updated_at DATETIME DEFAULT CURRENT_TIMESTAMP
)
```

**promotion_config_v2 - 优惠配置表**
```sql
CREATE TABLE promotion_config_v2 (
    id INTEGER PRIMARY KEY,
    product_id INTEGER NOT NULL,
    mode VARCHAR(20) NOT NULL DEFAULT 'credit',
    trigger_qty INTEGER NOT NULL DEFAULT 10,  -- 买 N
    gift_qty INTEGER NOT NULL DEFAULT 0,      -- 赠 M
    discount_rate FLOAT DEFAULT 100.0,
    is_active INTEGER DEFAULT 1,
    description VARCHAR(200),
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
    updated_at DATETIME DEFAULT CURRENT_TIMESTAMP
)
```

**核心类**:
```python
# 策略模式实现
class DiscountStrategy(ABC):
    """优惠策略基类"""
    
class CreditDiscountStrategy(DiscountStrategy):
    """先用后付：集采优惠，无优惠"""
    
class PrepaidDiscountStrategy(DiscountStrategy):
    """先付后用：买 N 赠 M 优惠"""

class DiscountContext:
    """优惠策略上下文"""
```

**优惠计算逻辑**:
- **信用模式**: 标准价格，无优惠
- **预付模式**: 买 N 赠 M，根据历史交易记录计算免费数量

**示例**:
```python
# 买 10 赠 1，购买 25 个
# 周期 = 11，25 / 11 = 2 个周期...余 3
# 免费数量 = 2（第 11 个和第 22 个）
# 付费数量 = 23
```

## 🔄 与原系统对比

### Before（当前实现）

```
┌────────────────────────────────────────────────┐
│  先用后付流程                                   │
│  用户领取 → 创建 Transaction → 记录应付 → 结算  │
└────────────────────────────────────────────────┘

┌────────────────────────────────────────────────┐
│  先付后用流程                                   │
│  创建订单 → 确认收款 → 领取 → 扣减订单余额      │
└────────────────────────────────────────────────┘

问题:
- 数据冗余：Transaction 和 PrepaidOrder 都记录预付信息
- 逻辑分散：领取逻辑在多处实现
- 体验割裂：用户需要理解两套流程
```

### After（统一架构）

```
┌────────────────────────────────────────────────┐
│  统一领取流程                                   │
│  选择产品 → 输入数量 → 自动扣款 → 创建交易      │
│      ↓                                          │
│  优先扣预付（优惠）→ 不足扣信用（标准价）       │
└────────────────────────────────────────────────┘

优势:
✅ 统一领取逻辑
✅ 简化用户认知
✅ 后台差异化处理
✅ 高扩展性架构
```
