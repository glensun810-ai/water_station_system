## 一、总体设计思路

### 1.1 核心原则

1. **保留已实现功能**：水站管理模块 (`Service_WaterManage/frontend/admin.html` 和 `index.html`) 已实现的功能全部保留
2. **数据源统一切换**：
   - 办公室数据：使用 `portal/admin/offices.html` 的后端API（`/api/offices`）
   - 用户数据：使用 `portal/admin/users.html` 的后端API（`/api/users`）
3. **功能结构不变**：保持原有水站管理功能模块划分，仅更换数据来源
4. **渐进式演进**：不破坏现有功能，逐步切换数据源

### 1.2 数据源对应关系

| 数据项 | 原数据来源 | 新数据来源 | 说明 |
|--------|-----------|-----------|------|
| 办公室列表 | `/api/offices` (水站专用) | `/api/offices` (统一) | 使用portal的办公室管理API |
| 用户列表 | `/api/users` (水站专用) | `/api/users` (统一) | 使用portal的用户管理API |
| 产品列表 | `/api/products` | `/api/products` | 保持不变 |
| 领水记录 | `/api/water/pickups` | `/api/water/pickups` | 保持不变 |
| 结算记录 | `/api/settlements` | `/api/settlements` | 保持不变 |
| 预付账户 | `/api/office-accounts` | `/api/office-accounts` | 保持不变 |
| 仪表盘统计 | `/api/dashboard/water` | `/api/dashboard/water` | 保持不变 |

## 六、数据源切换详细方案

### 6.1 办公室数据

**原API**：`Service_WaterManage` 内部API
**新API**：`portal/admin/offices.html` 使用的 `/api/offices`

```javascript
// 改造前 (Service_WaterManage/frontend/admin.html)
async function loadOffices() {
    const res = await fetch('/api/offices');  // 水站专用接口
    const data = await res.json();
    // 处理水站专用格式
}

// 改造后 (portal/admin/water/pickups.html)
async function loadOffices() {
    const res = await fetch('/api/offices');  // 统一接口
    const data = await res.json();
    // 处理统一格式
}
```

**兼容性处理**：
1. 确保 `/api/offices` 返回格式兼容
2. 添加字段映射（如需要）
3. 保持向后兼容

### 6.2 用户数据

**原API**：`Service_WaterManage` 内部API
**新API**：`portal/admin/users.html` 使用的 `/api/users`

```javascript
// 改造前 (Service_WaterManage/frontend/admin.html)
async function loadUsers() {
    const res = await fetch('/api/users/list');  // 水站专用
}

// 改造后
async function loadUsers() {
    const res = await fetch('/api/users');  // 统一
}
```

### 6.3 API接口统一清单

| 模块 | 原API | 新API | 状态 |
|------|-------|-------|------|
| 办公室列表 | `/api/offices` | `/api/offices` | ✅ 相同 |
| 办公室详情 | `/api/offices/{id}` | `/api/offices/{id}` | ✅ 相同 |
