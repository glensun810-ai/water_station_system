## 一、项目组织架构与角色分工

## 三、开发流程规范

### 3.1 开发流程总览

```
开发流程：

需求确认 → 技术设计 → 任务分配 → 开发执行 → 代码审查 → 测试验证 → 验收发布
```

### 3.2 需求确认流程

**参与者**：产品经理、业务代表、项目经理、技术架构师

**流程步骤**：

1. **需求收集**（产品经理主导）
   - 产品经理收集业务需求（访谈业务代表）
   - 编写需求文档（docs/requirements/[迭代号]_requirements.md）
   - 需求文档包含：
     - 业务背景
     - 功能需求
     - 非功能需求（性能、安全）
     - 业务规则
     - 验收标准

2. **需求评审**（项目经理组织）
   - 时间：迭代开始前1天
   - 参与者：产品经理、业务代表、技术架构师、开发组长、测试组长
   - 评审内容：
     - 需求合理性
     - 技术可行性
     - 工作量估算
     - 优先级排序
   - 产出：需求评审报告、任务清单

3. **需求确认**（业务代表签字）
   - 业务代表确认需求文档
   - 签字确认（电子签名或邮件确认）
   - 需求文档归档（Git版本管理）

### 3.3 技术设计流程

**参与者**：技术架构师、开发组长、开发工程师

**流程步骤**：

1. **架构设计**（技术架构师主导）
   - 设计系统架构（模块划分、依赖关系）
   - 设计数据库表结构（ER图、字段定义）
   - 设计API接口（接口文档、数据格式）
   - 编写架构设计文档（docs/design/[迭代号]_architecture.md）

2. **技术方案评审**（技术架构师组织）
   - 时间：迭代第1天下午
   - 参与者：技术架构师、开发组长、项目经理
   - 评审内容：
     - 架构合理性
     - 技术选型合理性
     - 性能影响评估
     - 安全风险评估
   - 产出：技术方案评审报告

3. **任务分解**（开发组长主导）
   - 将需求分解为开发任务
   - 每个任务粒度：0.5-2天工作量
   - 分配给开发工程师
   - 编写任务清单（docs/tasks/[迭代号]_task_list.md）

### 3.4 开发执行流程

**参与者**：开发工程师、开发组长

**流程步骤**：

1. **任务领取**（开发工程师）
   - 从任务清单领取任务
   - 更新任务状态（进行中）
   - 在GitLab创建分支（feature/[任务名称]）

2. **代码开发**（开发工程师）
   - 编写代码（遵循编码规范）
   - 编写单元测试（pytest）
   - 编写API文档（Swagger注释）
   - 本地测试验证

3. **代码提交**（开发工程师）
   - 代码提交到GitLab分支
   - 提交信息规范：`[SprintX] TaskName: Description`
   - 示例：`[Sprint1] ModuleSeparation: 将api_meeting.py迁移到Service_MeetingRoom`

4. **代码审查**（开发组长 + 技术架构师）
   - GitLab发起Merge Request
   - 开发组长审查代码（代码规范、逻辑正确性）
   - 技术架构师审查架构（架构一致性、性能影响）
   - 审查通过后合并到develop分支
   - 审查不通过则打回修改（附审查意见）

**代码审查标准**：

| 审查项 | 审查标准 | 审查人 |
|--------|---------|--------|
| 代码规范 | 遵循PEP8规范，使用Ruff格式化 | 开发组长 |
| 逻辑正确性 | 业务逻辑正确，边界条件处理完善 | 开发组长 |
| 单元测试 | 单元测试覆盖率≥80%，测试用例合理 | 开发组长 |
| API文档 | Swagger注释完整，参数说明清晰 | 开发组长 |
| 架构一致性 | 符合架构设计，模块职责清晰 | 技术架构师 |
| 性能影响 | 无明显性能问题（查询优化、缓存使用） | 技术架构师 |
| 安全风险 | 无安全漏洞（SQL注入、敏感信息泄露） | 技术架构师 |

### 3.5 开发编码规范

**Python编码规范**：

1. **命名规范**
   - 类名：大驼峰（UserService）
   - 函数名：小驼峰（getUserBalance）
   - 变量名：小驼峰（userId）
   - 常量名：全大写下划线（MAX_RETRY_COUNT）

2. **注释规范**
   - 文件头注释：说明文件用途、作者、日期
   - 函数注释：说明功能、参数、返回值
   - 关键逻辑注释：说明复杂逻辑的处理思路

3. **代码格式**
   - 使用Ruff自动格式化
   - 缩进：4空格
   - 最大行宽：120字符
   - 导入顺序：标准库 → 第三方库 → 本地库

4. **错误处理**
   - 使用统一异常类（BusinessError、ValidationError）
   - 不使用裸except，明确异常类型
   - 异常信息记录到日志，不暴露给前端

5. **日志记录**
   - 使用统一日志框架（utils/logger.py）
   - 关键操作记录INFO日志
   - 异常记录ERROR日志
   - 日志格式：JSON结构化

**Vue.js编码规范**：

1. **组件命名**
   - 组件文件名：大驼峰（WaterPickup.vue）
   - 组件注册名：大驼峰（WaterPickup）

2. **组件结构**
   - template → script → style
   - 组件属性顺序：props → data → computed → methods → lifecycle

3. **代码格式**
   - 使用Prettier自动格式化
   - 缩进：2空格
   - 最大行宽：100字符

## 四、测试流程规范

### 4.1 测试流程总览

```
测试流程：

测试计划 → 测试设计 → 测试执行 → Bug修复 → 回归测试 → 测试报告 → 验收测试
```

### 4.2 测试计划流程

**参与者**：测试组长、项目经理、开发组长

**流程步骤**：

1. **测试计划编写**（测试组长）
   - 时间：迭代第1天
   - 内容：
     - 测试范围（测试哪些功能）
     - 测试类型（功能测试、性能测试、安全测试）
     - 测试环境（测试数据库、测试服务器）
     - 测试工具（pytest、Playwright）
     - 测试进度（测试时间安排）
   - 文档：docs/tests/[迭代号]_test_plan.md

2. **测试计划评审**（项目经理组织）
   - 时间：迭代第1天下午
   - 参与者：测试组长、项目经理、开发组长
   - 评审内容：
     - 测试范围完整性
     - 测试时间合理性
     - 测试环境准备情况
   - 产出：测试计划评审报告

### 4.3 测试设计流程

**参与者**：测试工程师、测试组长

**流程步骤**：

1. **测试用例设计**（测试工程师）
   - 时间：迭代第2-3天
   - 内容：
     - 功能测试用例（正常流程、异常流程、边界条件）
     - 性能测试用例（响应时间、并发性能）
     - 安全测试用例（SQL注入、权限验证）
   - 文档：docs/tests/[迭代号]_test_cases.xlsx

2. **测试数据准备**（测试工程师）
   - 准备测试数据（测试数据库初始化）
   - 测试数据脚本：tests/fixtures/test_data.sql
   - 测试数据包含：
     - 测试用户（内部用户、外部用户）
     - 测试产品（水站产品、会议室）
     - 测试订单（充值记录、预约记录）

3. **测试环境搭建**（运维工程师 + 测试工程师）
   - 搭建测试服务器（独立测试环境）
   - 配置测试数据库（test_waterms.db、test_meeting.db）
   - 配置测试配置文件（.env.test）

### 4.4 测试执行流程

**参与者**：测试工程师、开发工程师

**流程步骤**：

1. **功能测试执行**（测试工程师）
   - 时间：迭代第11-12天
   - 执行测试用例（按测试用例逐项测试）
   - 记录测试结果（通过/失败）
   - 记录bug（发现bug提交到Bug管理系统）

2. **Bug提交**（测试工程师）
   - 使用GitLab Issues提交bug
   - Bug标题：`[Bug] [SprintX] BugDescription`
   - Bug内容：
     - Bug描述（现象、预期结果、实际结果）
     - Bug优先级（P0/P1/P2/P3）
     - Bug截图或日志
     - 重现步骤
     - 测试环境信息

3. **Bug修复**（开发工程师）
   - 开发工程师领取bug任务
   - 修复bug并提交代码
   - 提交修复说明（GitLab Issues回复）
   - 通知测试工程师回归测试

4. **回归测试**（测试工程师）
   - 重新测试bug修复后的功能
   - 测试相关功能是否受影响
   - 记录回归测试结果

5. **性能测试**（测试工程师）
   - 使用工具：Locust或JMeter
   - 测试场景：
     - API响应时间（P95 < 500ms）
     - 并发性能（100并发用户）
     - 数据库查询性能（慢查询检查）
   - 记录性能测试结果

6. **安全测试**（测试工程师）
   - 测试场景：
     - SQL注入测试（输入特殊字符）
     - XSS攻击测试（注入恶意脚本）
     - 权限绕过测试（未授权访问）
     - 敏感信息泄露测试（错误信息暴露）
   - 记录安全测试结果

### 4.5 测试报告流程

**参与者**：测试组长、项目经理

**流程步骤**：

1. **测试报告编写**（测试组长）

     - 测试概要（测试范围、测试时间）
     - 测试结果（通过率、Bug统计）
     - Bug分析（Bug分布、Bug原因）
     - 性能测试结果
     - 安全测试结果
     - 测试结论（通过/不通过）
   - 文档：docs/tests/[迭代号]_test_report.md

2. **测试报告评审**（项目经理组织）
   - 时间：迭代第13天下午
