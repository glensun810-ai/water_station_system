# 用户认证流程彻底修复报告

## 问题分析

### 根本原因
用户管理页面反复出现"未登录或登录已过期"错误的根本原因：
1. **登录页面使用了错误的API路径** - `/api/user/login`（不存在）而非 `/api/v1/system/auth/login`
2. **用户管理页面没有登录状态检查** - 页面初始化时直接调用API，不检查localStorage中是否有token
3. **缺少统一的认证处理机制** - 每个API调用都重复处理token获取和401错误
4. **登录参数不匹配** - 使用 `{name: ...}` 而API期望 `{username: ...}`

### 问题影响
- 用户登录成功后，token被保存到localStorage
- 但用户管理页面在加载时，如果localStorage中没有token（或token过期），直接调用API导致401错误
- 401错误后，页面跳转到登录页面，但登录页面可能也有问题
- 整个认证流程链路断裂，导致反复提示"未登录或登录已过期"

## 修复方案

### 修复策略
采用**首席架构师级别的彻底重构**，建立统一的认证处理机制：

1. **统一认证检查函数** `checkAuth()`
   - 检查localStorage中是否有token
   - 如果没有token，立即跳转到登录页面（带redirect参数）
   - 在页面初始化时调用，防止无效API请求

2. **统一API错误处理函数** `handleApiError(response)`
   - 处理401错误（token过期）
   - 清除localStorage中的token和userInfo
   - 跳转到登录页面（带redirect参数）

3. **统一认证头获取函数** `getAuthHeaders()`
   - 自动从localStorage获取token
   - 构造标准Authorization header
   - 简化API调用代码

- 在 `onMounted` 中添加登录状态检查
- 替换所有API调用中的手动token获取和401处理

**修改位置：**
- Line 959-982: 添加三个统一认证处理函数
- Line 1469-1473: `onMounted` 中添加登录检查
- 所有API调用函数：统一使用 `getAuthHeaders()` 和 `handleApiError()`

### 代码优化

**优化前（重复代码）：**
```javascript
const token = localStorage.getItem('token');
const headers = { 'Content-Type': 'application/json' };
if (token) headers['Authorization'] = `Bearer ${token}`;

const response = await fetch(url, { headers });
if (response.status === 401) {
    alert('未登录或登录已过期，请重新登录');
    window.location.href = '/portal/admin/login.html';
}
```

**优化后（统一处理）：**
```javascript
if (!checkAuth()) return;

const response = await fetch(url, { headers: getAuthHeaders() });
if (handleApiError(response)) return;
```
