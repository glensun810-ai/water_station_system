## 一、当前业务流程分析

### 1.1 现有流程概览

**当前流程图**:
```
用户领水 → 记录生成(pending状态)
         ↓ (??? 申请环节缺失或不明确)
         (用户申请结算???)
         ↓ (前端显示applied状态，但后端逻辑混乱)
         管理员批量确认 → 直接改为settled
         ↓
         结算完成(缺少凭证和审计)
```

**流程问题识别**:
1. ❌ 申请环节不明确：用户何时、如何申请结算？
2. ❌ 状态流转混乱：pending/applied/settled的定义不清晰
3. ❌ 缺少审核环节：管理员直接批量确认，缺少审核步骤
4. ❌ 缺少收款核验：确认收款没有金额核对
5. ❌ 缺少凭证生成：没有收据、结算单等凭证

## 二、新业务流程设计

### 2.1 核心业务流程

#### 流程1: 领水记录生成流程

**流程图**:
```
用户扫码领水
    ↓
选择产品和数量
    ↓
系统自动计算金额(含优惠)
    ↓
生成领水记录
    ↓
状态: pending(待付款)
    ↓
关联办公室账户(扣减预留数量)
```

**关键节点**:
1. **领水时间**: 记录准确的领取时间
2. **产品选择**: 只显示库存>0的产品
3. **价格计算**: 自动应用买N赠M优惠
4. **状态初始化**: 统一设置为`pending`
5. **账户关联**: 自动关联用户所属办公室

**数据记录**:
```python
OfficePickup {
    id: 领水记录ID,
    office_id: 办公室ID,
    office_name: 办公室名称,
    product_id: 产品ID,
    product_name: 产品名称,
    specification: 规格,
    quantity: 数量,
    unit_price: 单价,
    total_amount: 总金额(含优惠),
    pickup_person: 领取人,
    pickup_person_id: 领取人ID,
    pickup_time: 领取时间,
    
    # 新增字段
    settlement_status: 'pending',  # 统一状态字段
    payment_mode: 'credit',  # 付款模式: credit(记账)/cash(现金)/transfer(转账)
    
    # 审计字段
    created_at: 创建时间,
    created_by: 创建人ID,
    updated_at: 更新时间,
    is_deleted: 软删除标记
}
```

#### 流程2: 结算申请流程

**流程图**:
```
办公室负责人查看待付款记录
    ↓
选择要结算的记录
    ↓
点击"申请结算"按钮
    ↓
系统生成结算申请单
    ↓
状态: applied(已申请待审核)
    ↓
通知管理员审核
```

**申请条件校验**:
1. ✅ 只有办公室负责人可以申请
2. ✅ 只能申请本办公室的记录
3. ✅ 只能申请pending状态的记录
4. ✅ 最少选择1条记录
5. ✅ 支持批量申请多条记录

**结算申请单数据结构**:
```python
SettlementApplication {
    id: 申请单ID,
    application_no: 申请单编号(格式: SA-YYYYMMDD-NNNN),
    
    office_id: 办公室ID,
    office_name: 办公室名称,
    office_room_number: 房间号,
    
    # 申请信息
    applicant_id: 申请人ID,
    applicant_name: 申请人姓名,
    applied_at: 申请时间,
    
    # 结算信息
    pickup_ids: 关联的领水记录ID列表,
    record_count: 记录数量,
    total_amount: 结算总金额,
    
    # 状态管理
    status: 'applied',  # pending/applied/approved/confirmed/rejected/cancelled
    
    # 审核信息
    reviewer_id: 审核人ID,
    reviewer_name: 审核人姓名,
    reviewed_at: 审核时间,
    review_note: 审核备注,
    
    # 收款信息
    confirmer_id: 确认人ID,
    confirmer_name: 确认人姓名,
    confirmed_at: 确认时间,
    payment_method: 收款方式,
    payment_amount: 实际收款金额,
    payment_evidence: 收款凭证(URL),
    
    # 结算周期
    settlement_period: 结算周期(如: 2026-04),
    settlement_type: 结算类型(monthly/adhoc),
    
    # 备注
    note: 申请备注,
    
    # 审计字段
    created_at: 创建时间,
    updated_at: 更新时间,
    is_deleted: 软删除标记
}
```

**申请单编号规则**:
```
格式: SA-YYYYMMDD-NNNN
示例: SA-20260408-0001

解释:
- SA: Settlement Application(结算申请)
- YYYYMMDD: 申请日期(2026年4月8日)
- NNNN: 当日流水号(从0001开始)
```

#### 流程3: 结算审核流程

**流程图**:
```
管理员收到结算申请通知
    ↓
查看申请详情(领水记录、金额、办公室)
    ↓
审核决策分支
    ├─ 通过 → 状态: approved(已审核待收款)
    │           ↓
    │       等待办公室付款
    │           ↓
    │       收款确认流程
    │
    └─ 拒绝 → 状态: rejected(已拒绝)
              ↓
          记录退回pending状态
          ↓
          通知申请人
```

**审核要点**:
1. **金额核对**: 检查申请金额是否准确
2. **记录验证**: 检查领水记录是否真实
3. **权限确认**: 确认申请人是否有权限
4. **异常检查**: 检查是否有异常记录(如大量领水)

**审核通过条件**:
- ✅ 申请金额计算正确
- ✅ 所有领水记录属于同一办公室
- ✅ 申请人是该办公室负责人
- ✅ 没有异常或争议记录
- ✅ 所有记录状态为pending

**审核拒绝场景**:
- ❌ 金额计算错误
- ❌ 包含其他办公室的记录
- ❌ 申请人无权限
- ❌ 存在争议记录
- ❌ 记录状态异常

---

#### 流程4: 收款确认流程

**流程图**:
```
办公室负责人提交付款
    ↓
上传收款凭证(可选)
    ↓
管理员核对收款信息
    ↓
收款信息录入
    ├─ 收款方式
    ├─ 收款金额
    ├─ 收款时间
    └─ 收款凭证
    ↓
确认收款按钮
    ↓
状态: confirmed(已收款待结清)
    ↓
生成收款收据
    ↓
完成结清
    ↓
状态: settled(已结清)
    ↓
领水记录状态同步更新为settled
```

**收款确认要点**:
1. **收款方式选择**:
   - cash: 现金收款
   - transfer: 银行转账
   - alipay: 支付宝
   - wechat: 微信支付
   - credit: 记账(月结)

2. **收款金额核对**:
   - 申请金额 vs 实际收款金额
   - 差额记录(如有)
   - 部分收款处理(如有)

3. **收款凭证上传**:
   - 支持图片上传(PNG/JPG)
   - 支持PDF凭证
   - 可选(非必填)

4. **收据生成**:
   - 自动生成收据编号
   - 包含收款详情
   - 支持打印和导出

#### 流程5: 月度结算流程(新增)

**流程图**:
```
月底自动触发(或手动触发)
    ↓
系统筛选本月所有pending记录
    ↓
按办公室分组汇总
    ↓
生成月度结算单
    ↓
状态: draft(草稿)
    ↓
管理员审核月度结算单
    ↓
审核通过 → 状态: pending_payment(待付款)
    ↓
批量发送付款通知
    ↓
各办公室逐个付款
    ↓
逐个确认收款
    ↓
全部收款完成 → 状态: settled(已结清)
    ↓
归档月度结算单
```

**月度结算单数据结构**:

### 2.2 状态流转规则

#### 状态定义表

**领水记录状态(OfficePickup.settlement_status)**:
```python
STATUS_PENDING = 'pending'      # 待付款(初始状态)
STATUS_APPLIED = 'applied'      # 已申请待审核
STATUS_APPROVED = 'approved'    # 已审核待收款
STATUS_CONFIRMED = 'confirmed'  # 已收款待结清
STATUS_SETTLED = 'settled'      # 已结清(终态)
STATUS_REJECTED = 'rejected'    # 已拒绝(异常态)
STATUS_CANCELLED = 'cancelled'  # 已取消(异常态)
```

**结算申请单状态(SettlementApplication.status)**:
```python
STATUS_DRAFT = 'draft'          # 草稿(未提交)
STATUS_APPLIED = 'applied'      # 已申请待审核
STATUS_APPROVED = 'approved'    # 已审核待收款
STATUS_CONFIRMED = 'confirmed'  # 已收款待结清
STATUS_SETTLED = 'settled'      # 已结清(终态)
STATUS_REJECTED = 'rejected'    # 已拒绝
STATUS_CANCELLED = 'cancelled'  # 已取消
```

---

#### 状态流转规则表

**领水记录状态流转**:
```
pending → applied: 用户申请结算
applied → approved: 管理员审核通过
approved → confirmed: 管理员确认收款
confirmed → settled: 系统自动结清(同步申请单状态)

异常流转:
applied → rejected: 管理员审核拒绝 → 回退到pending
approved → cancelled: 申请取消 → 回退到pending
```

**结算申请单状态流转**:
```
draft → applied: 提交申请
applied → approved: 审核通过
approved → confirmed: 确认收款
confirmed → settled: 完成结清

异常流转:
applied → rejected: 审核拒绝 → 申请单关闭，记录回退
approved → cancelled: 申请取消 → 申请单关闭，记录回退
settled → cancelled: 结清作废(特殊操作，需审批)
```

---

#### 状态流转约束

**规则1: 单向流转原则**
- ✅ 正常流程: pending → applied → approved → confirmed → settled
- ❌ 禁止反向流转(已结清不能回退到待付款)

**规则2: 异常回退原则**
- ✅ 拒绝和取消可以回退到pending状态
- ✅ 回退操作必须记录原因和操作人
- ❌ 已收款记录不允许回退

**规则3: 状态一致性原则**
- ✅ 结算申请单状态必须与关联记录状态一致
- ✅ 记录状态变更必须同步更新申请单状态
- ❌ 禁止单独修改记录状态而不更新申请单

**规则4: 终态锁定原则**
- ✅ settled状态为终态，不可修改
- ✅ 只有特殊审批才能作废已结清记录
- ❌ 禁止删除已结清记录

### 2.3 异常流程处理

#### 异常场景1: 部分收款

**场景描述**:
办公室申请结算金额¥800，实际只付款¥700。

**处理流程**:
```
管理员确认收款
    ↓
录入实际收款金额¥700
    ↓
标记差额¥100
    ↓
状态: partially_settled(部分结清)
    ↓
选择处理方式
    ├─ 继续追缴 → 保持状态，发送催款通知
    ├─ 作废差额 → 管理员审批后作废¥100，状态改为settled
    └─ 分期付款 → 设置分期计划，后续继续收款
```

**数据记录**:
```python
SettlementApplication {
    status: 'partially_settled',
    payment_amount: 700,  # 实际收款
    total_amount: 800,    # 申请金额
    difference_amount: 100,  # 差额
    difference_status: 'pending',  # pending/waived/installment
    difference_note: '办公室称本月预算不足，申请下月补缴'
}
```

---

#### 异常场景2: 争议记录

**场景描述**:
办公室对某条领水记录有争议，否认该记录真实性。

**处理流程**:
```
办公室负责人标记争议记录
    ↓
上传争议说明和证据
    ↓
记录状态: disputed(有争议)
    ↓
管理员审核争议
    ├─ 争议有效 → 记录作废，状态改为cancelled
    │               ↓
    │           从结算申请单中移除该记录
    │               ↓
    │           重新计算金额
    │
    └─ 争议无效 → 记录恢复正常，继续结算流程
                  ↓
              通知办公室负责人
```

---

#### 异常场景3: 误操作撤销

**场景描述**:
管理员误将pending记录直接改为settled(跳过申请环节)。

**处理流程**:
```
发现误操作
    ↓
管理员提交撤销申请
    ↓
超级管理员审批
    ├─ 通过 → 执行撤销操作
    │           ↓
    │       记录状态回退到pending
    │           ↓
    │       记录撤销原因和操作人
    │           ↓
    │       发送通知给办公室负责人
    │
    └─ 拒绝 → 撤销申请驳回，保持settled状态
              ↓
          记录拒绝原因
```

---

#### 异常场景4: 结清作废

**场景描述**:
已结清的记录需要作废(如:退款、系统错误等)。

**处理流程**:
```
发现需要作废
    ↓
管理员提交作废申请(必须说明原因)
    ↓
超级管理员审批
    ↓
审批通过 → 执行作废
    ├─ 生成退款单
    ├─ 记录状态改为cancelled
    ├─ 关联的结算申请单状态改为cancelled
    └─ 记录作废原因和审批人
    ↓
财务处理退款
    ↓
归档作废记录
```

### 2.4 批量操作流程

#### 批量操作类型定义

**操作1: 批量申请结算**
**操作人**: 办公室负责人  
**前置条件**:
- ✅ 只能选择本办公室的pending记录
- ✅ 至少选择1条记录
- ✅ 记录无异常或争议

**流程**:
```
选择多条pending记录
    ↓
点击"批量申请结算"
    ↓
预览申请详情(记录列表、金额汇总)
    ↓
确认申请
    ↓
生成结算申请单
    ↓
所有选中记录状态改为applied
```

---

**操作人**: 管理员

- ✅ 只能选择applied状态的申请单
- ✅ 申请单无异常标记
- ✅ 申请人是办公室负责人

**流程**:
```
选择多条applied申请单
    ↓
点击"批量审核通过"
    ↓
预览审核详情(申请单列表、总金额)
    ↓
确认审核
    ↓
所有申请单状态改为approved
    ↓
关联记录状态同步改为approved
```

---

**操作3: 批量确认收款**
**操作人**: 管理员  
**前置条件**:
- ✅ 只能选择approved或confirmed状态的申请单
- ✅ 已核对收款金额
- ✅ 已选择收款方式

**流程**:
```
选择多条approved申请单
    ↓
点击"批量确认收款"
    ↓
录入收款信息(统一收款方式和金额)
    ↓
预览确认详情(申请单列表、收款总额)

确认收款
    ↓
所有申请单状态改为settled
    ↓
关联记录状态同步改为settled
    ↓
批量生成收据
```

---

**操作4: 批量发送催款提醒**
**操作人**: 管理员  
**前置条件**:
- ✅ 只能选择pending或applied状态的记录
- ✅ 记录超过指定天数未结算(7/15/30天)

**流程**:
```
筛选逾期记录
    ↓
选择目标办公室或记录
    ↓
```

### 3.1 计算规则

#### 规则1: 金额计算规则

**优惠计算**:
```python
def calculate_total_amount(product, quantity):
    """
    计算总金额(含优惠)
    
    Args:
        product: 产品信息(含promo_threshold, promo_gift)
        quantity: 领取数量
    
    Returns:
        total_amount: 总金额
        free_quantity: 免费数量
    """
    if product.promo_threshold > 0:
        # 计算免费数量
        free_quantity = (quantity // product.promo_threshold) * product.promo_gift
        paid_quantity = quantity - free_quantity
        total_amount = paid_quantity * product.price
    else:
        free_quantity = 0
        total_amount = quantity * product.price
    
    return total_amount, free_quantity
```

**示例**:
- 产品: 18L桶装水，单价¥15，买10赠1
- 领取: 23桶
- 计算:
  - 免费数量 = (23 // 10) * 1 = 2桶
  - 付费数量 = 23 - 2 = 21桶
  - 总金额 = 21 * 15 = ¥315
