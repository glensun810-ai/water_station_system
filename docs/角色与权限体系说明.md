# AI产业集群空间服务系统 - 角色与权限体系说明

## 核心业务概念

### 办公室（Office）
- **核心单元**：系统以"办公室"为核心管理单元，而非部门
- **业务逻辑**：每个办公室独立管理用户、账户、服务
- **负责人**：每个办公室可配置负责人，自动成为该办公室的管理员

### 用户所属
- `department` 字段：实际表示用户所属的**办公室名称**（非部门）
- 该字段在用户管理中表示办公室绑定关系

---

## 角色体系

| 角色 | 标识 | 权限范围 | 数量限制 |
|------|------|----------|----------|
| 超级管理员 | `super_admin` | 系统最高权限，管理所有办公室和用户 | ≥ 1 |
| 系统管理员 | `admin` | 管理系统配置、用户、办公室 | 多个 |
| 办公室管理员 | `office_admin` | 管理特定办公室的用户和服务 | 每个办公室可多个 |
| 普通用户 | `user` | 使用服务的基本用户 | 不限 |

### 权限说明

#### 超级管理员 (super_admin)
- ✅ 创建/删除系统管理员
- ✅ 创建/删除办公室
- ✅ 管理所有办公室的用户
- ✅ 系统配置管理
- ✅ 无法被删除或停用（最后一个）

#### 系统管理员 (admin)
- ✅ 创建/删除办公室
- ✅ 管理所有办公室的用户
- ✅ 查看系统统计数据
- ❌ 无法管理超级管理员

#### 办公室管理员 (office_admin)
- ✅ 审核本办公室新用户注册
- ✅ 管理本办公室用户
- ✅ 查看本办公室统计数据
- ✅ 批量导入本办公室用户
- ❌ 无法管理其他办公室

#### 普通用户 (user)
- ✅ 使用水站服务
- ✅ 使用会议室预订服务
- ✅ 查看个人账户
- ❌ 无管理权限

## 办公室负责人自动创建账号机制

### 触发条件
1. 创建/编辑办公室时填写了负责人姓名
2. 勾选"同时创建负责人登录账号"
3. 系统检测该用户名不存在

### 创建规则
```
用户名：负责人姓名
默认密码：123456
角色：办公室管理员 (office_admin)
所属办公室：该办公室名称
权限：管理该办公室的用户和服务
```

### 同名用户处理
- 系统实时检查用户名是否存在
- 存在同名用户时：
  - 显示已有用户信息（角色、所属办公室）
  - 禁用"创建账号"选项
  - 提示前往用户管理手动处理

### 示例流程

```
场景：创建"技术部"办公室，负责人"张三"

步骤1：填写办公室信息
┌─────────────────────────┐
│ 办公室名称：技术部       │
│ 门号：301               │
│ 负责人：张三            │
└─────────────────────────┘

步骤2：系统自动检查
→ 检测用户名"张三"不存在

步骤3：勾选创建账号
☑ 同时创建负责人登录账号

系统提示：

✓ 权限：审核用户、管理办公室用户

步骤4：提交创建
→ 创建办公室成功
→ 创建用户成功
→ 添加到办公室管理员关联表

结果提示：
✅ 办公室创建成功！

已自动创建办公室管理员账号：
• 用户名：张三
• 默认密码：123456
• 所属办公室：技术部
• 角色：办公室管理员
• 权限：审核该办公室新用户注册、管理该办公室用户、查看该办公室统计数据

```

## 数据表关系

### 用户表 (users)
```
id          - 用户ID
name        - 用户名（唯一）
department  - 所属办公室名称
role        - 角色
is_active   - 是否启用
```

### 办公室表 (office)
```
id                  - 办公室ID
name                - 办公室名称
leader_name         - 负责人姓名
primary_admin_id    - 主要管理员ID（用户表外键）
```

### 办公室管理员关联表 (office_admin_relations)
```
id          - 关联ID
office_id   - 办公室ID
user_id     - 管理员用户ID
is_primary  - 是否主要管理员
role_type   - 管理员类型（1=负责人，2=行政对接人，3=其他）
```

## 权限保护机制

### 超级管理员保护
1. **删除保护**：禁止删除最后一个超级管理员
2. **停用保护**：禁止停用最后一个活跃的超级管理员
3. **降级保护**：禁止降级最后一个超级管理员的角色
4. **批量操作保护**：批量操作时检查超级管理员数量

### 前端防护
- 显示超级管理员特殊标识 🛡️
- 危险操作需要二次确认
- 提升为超级管理员需要输入"确认"

### 后端防护
- API层面硬性阻断危险操作
- 返回明确的错误提示
- 保护系统不被锁死

## 常见场景

### 场景1：新办公室入驻
1. 超级管理员创建办公室
2. 填写负责人信息
3. 自动创建办公室管理员账号
4. 负责人登录系统修改密码
5. 负责人开始管理该办公室用户

### 场景2：负责人变更
1. 编辑办公室信息
2. 修改负责人姓名
3. 可选择创建新的管理员账号
4. 旧管理员可保留或调整角色

### 场景3：办公室合并
1. 将用户迁移到新办公室
2. 调整办公室管理员权限
3. 停用或删除旧办公室

## 技术实现要点

1. **实时用户名检查**：输入负责人姓名时500ms延迟检查
2. **事务保护**：创建办公室和用户使用数据库事务
3. **关联表维护**：自动维护office_admin_relations表
4. **密码加密**：使用bcrypt加密默认密码
5. **响应信息**：详细返回创建结果和账号信息

---

## 注意事项

1. **密码安全**：默认密码123456，首次登录强制修改
2. **权限隔离**：办公室管理员只能管理本办公室
3. **数据一致性**：删除办公室前需处理关联用户
4. **操作日志**：关键操作记录日志
5. **用户体验**：清晰的操作提示和引导
